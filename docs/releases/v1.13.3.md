## Refatoracao: radioService.ts

### O problema (v1.13.1-v1.13.2)

As v1.13.1 e v1.13.2 tentaram corrigir o lock screen via patches no codigo
nativo do expo-audio (`withMetadataFix.js`). Os patches eram corretos mas
possivelmente nao eram compilados pelo Gradle que usava build cache das
compilacoes anteriores (builds mostravam "553 up-to-date").

Alem disso, o codigo JS usava dois caminhos nativos:
1. `setActiveForLockScreen(true, meta)` → chamava `setPlayerOptions()` no nativo
2. `updateLockScreenMetadata(meta)` → chamava `updateMetadataInternal()` no nativo

O caminho 2 (`updateMetadataInternal`) tinha o bug: delegava o rebuild da
notificacao ao callback de `loadArtworkFromUrl()`, que nao disparava quando
`java.net.URL.equals()` considerava URLs iguais (ignora fragments).

O caminho 1 (`setPlayerOptions`) NAO tem este bug — chama
`postOrStartForegroundNotification()` incondicionalmente.

### A solucao

**radioService.ts**: Usar SEMPRE `setActiveForLockScreen(true, meta)`.
Quando o player ja e o `currentPlayer` no nativo, `setPlayerOptions()` faz
update leve (sem recriar MediaSession):
1. Actualiza `currentMetadata`
2. Chama `loadArtworkFromUrl()` (com patch: sem URL check + timeout 8s)
3. Chama `postOrStartForegroundNotification()` incondicionalmente

Removido:
- `lockScreenSessionActive` flag (ja nao ha dois caminhos)
- `updateLockScreenMetadata()` nunca mais e chamado

**Build**: `./gradlew clean` antes do build para forcar recompilacao do
AudioControlsService.kt patchado.

### Ficheiros alterados

- `src/services/radioService.ts` — refatoracao do `updateLockScreen()`
- `app.json` / `package.json` — bump versao
- `plugins/withMetadataFix.js` — mantido (remove URL check + timeout)
