# Release v1.13.2 (Build 61)

**Data**: 2026-04-20
**versionName**: 1.13.2
**versionCode**: 61 (Android)

---

## Resumo da Release

Fix definitivo para o lock screen metadata que ficava travado no titulo/artista/artwork
de uma musica anterior. O fix da v1.13.1 (fragment cache-buster) nao funcionava
porque `java.net.URL.equals()` **ignora fragments por design** — o lado nativo
continuava a considerar a URL "igual" e pulava o rebuild da notificacao.

---

## Correcao: Lock screen metadata travado

### Causa raiz (descoberta na v1.13.2)

O bug esta no `AudioControlsService.kt` do expo-audio, metodo `updateMetadataInternal()`:

```kotlin
currentMetadata?.artworkUrl?.let {
    loadArtworkFromUrl(it) { bitmap ->
        currentArtwork = bitmap
        postOrStartForegroundNotification(...)  // unico lugar que reconstroi
    }
} ?: postOrStartForegroundNotification(...)
```

`loadArtworkFromUrl()` compara URLs com `java.net.URL.equals()`, que ignora o
fragment (`#timestamp`). Quando a artwork URL e "igual" (mesmo ficheiro de logo
com fragment diferente), o metodo retorna **sem chamar o callback**. Como
`postOrStartForegroundNotification()` so e chamado DENTRO do callback, a
notificacao nunca reconstroi — titulo, artista e artwork ficam congelados.

### Fix: Plugin `withMetadataFix.js`

Novo Expo config plugin que patcha `AudioControlsService.kt` no prebuild:

**FIX 1** — `updateMetadataInternal`: chamada incondicional a
`postOrStartForegroundNotification()` apos atualizar `currentMetadata`.
A notificacao reconstroi imediatamente com metadata novo + artwork existente.
Se a artwork tambem mudar, um segundo rebuild vem quando o download termina.

**FIX 2** — `loadArtworkFromUrl`: timeout de 8s (connectTimeout + readTimeout)
no `url.openConnection()`. O codigo original nao tinha timeout algum — podia
travar indefinidamente em background ou rede lenta.

### Ficheiros alterados

- `plugins/withMetadataFix.js` (NOVO) — plugin que aplica os dois fixes
- `app.json` — registo do plugin + bump de versao

### Porque o fix da v1.13.1 nao bastou

A v1.13.1 adicionou `#${Date.now()}` ao fallback logo URL, assumindo que
`URL.equals()` compararia fragments. Mas a spec de `java.net.URL` diz
explicitamente: *"the ref field is not taken into consideration"*.
O fragment era ignorado, a URL era "igual", e o callback nunca disparava.
