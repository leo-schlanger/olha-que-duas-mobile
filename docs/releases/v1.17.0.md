# v1.17.0 — Native background metadata polling

**Build 71 | 2026-04-30**

## Problema resolvido

### Notificacao/lock screen nao atualizava em background
- **Causa raiz:** O Android suspende completamente a thread JS do React Native quando o app vai para background, mesmo com foreground service ativo. O ExoPlayer (nativo) continua tocando audio, mas todo o pipeline JS (SSE, polling, callbacks) para de executar. A notificacao ficava presa na ultima musica detectada antes do background.
- **Diagnostico:** Confirmado via logcat — zero logs `ReactNativeJS` em 90+ segundos de background, enquanto o foreground service (`MediaService`) continuava ativo e o audio tocava normalmente.

## Fix: Native metadata polling

O `MediaService` agora faz polling direto na API do AzuraCast, independente da thread JS:

- **pollMetadataFromApi()** — busca `/api/nowplaying/{shortcode}` a cada 10s no `artworkHandler` thread (Java thread nativa)
- **Parseia JSON** — extrai title, artist, artwork URL do payload do AzuraCast
- **Dedup nativo** — so chama `updateMetadata()` quando title/artist mudam
- **Artwork via URL remota** — o native side faz download do bitmap diretamente (fallback funcional, JS side continua preferindo file:// em foreground)
- **startMetadataPolling(url)** — iniciado ao ativar a MediaSession
- **stopMetadataPolling()** — parado ao desativar

### Resultado
- Notificacao atualiza titulo, artista e artwork em background em ~10s apos troca de musica
- JS side continua funcionando em foreground (polling mais rapido, artwork cache local)
- Native side e safety net para background (quando JS esta suspensa)
- Testado no emulador: 3 trocas de musica detectadas consecutivamente em background
