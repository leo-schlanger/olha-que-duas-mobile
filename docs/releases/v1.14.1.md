# Release v1.14.1 (Build 65)

**Data**: 2026-04-24
**versionName**: 1.14.1
**versionCode**: 65 (Android)

---

## Resumo da Release

Fix para artwork do lock screen que nao atualizava em background. Abordagem
hibrida: expo-audio gere o foreground service e notificacao (titulo/artista),
e o modulo nativo `expo-media-session` substitui o bitmap do artwork
directamente na notificacao — contornando o `loadArtworkFromUrl` quebrado
do expo-audio.

---

## Contexto (v1.14.0 → v1.14.1)

A v1.14.0 tentou substituir completamente a notificacao do expo-audio por
um modulo nativo custom com `MediaSessionCompat`. O foreground service do
modulo nao arrancou, resultando em:
- Sem notificacao
- Audio morre apos ~3 min em background (sem foreground service)
- Artwork nao actualiza

**Descoberta critica:** expo-audio PRECISA de `setActiveForLockScreen(true)`
para iniciar o seu foreground service. Sem isso, `shouldPlayInBackground: true`
so funciona por ~3 minutos (limitacao do Android OS).

---

## A solucao: notificacao hibrida

### Fase 1 — expo-audio (titulo + artista + foreground service)

`setActiveForLockScreen(true, meta)` faz o que sempre fez:
- Inicia o `AudioControlsService` como foreground service
- Cria notificacao `MediaStyle` com titulo, artista, controles play/pause
- Actualiza `MediaSession` metadata
- Chama `loadArtworkFromUrl()` para o bitmap (ESTE E O BUG — nao recarrega)

### Fase 2 — expo-media-session module (artwork override)

200ms apos a Fase 1, o modulo nativo:
1. Procura a notificacao activa no canal `expo_audio_channel`
2. Carrega o bitmap via `BitmapFactory.decodeFile()` (sem `URL.openConnection`)
3. Usa `Notification.Builder.recoverBuilder()` para reconstruir a notificacao
4. Substitui o `setLargeIcon()` com o bitmap correcto
5. Re-posta com o mesmo ID — substitui a notificacao do expo-audio

### Porque funciona

| Passo | Quem | O que faz |
|-------|------|-----------|
| T+0 | expo-audio | Rebuilds notificacao (titulo/artista novos, bitmap ANTIGO) |
| T+200ms | expo-media-session | Override: mesma notificacao, bitmap CORRECTO |

O expo-audio tenta carregar o bitmap no `loadArtworkFromUrl` thread, mas:
- Se `URL.equals()` retorna true → thread nao corre → nosso override e a unica fonte de bitmap
- Se `URL.equals()` retorna false → thread corre e re-posta bitmap → resultado correcto de qualquer forma

O override e uma SAFETY NET: funciona independentemente do comportamento do expo-audio.

## Alteracoes

### Modificado: `modules/expo-media-session/ExpoMediaSessionModule.kt`

- **Adicionado**: `overrideNotificationArtwork(artworkUri)` — procura notificacao por canal, carrega bitmap, re-posta
- **Adicionado**: `Notification.Builder.recoverBuilder()` para preservar titulo/artista/controles
- **Mantido**: WiFi lock (`acquireWifiLock/releaseWifiLock`)
- Bitmap escalado a max 512x512 para evitar `TransactionTooLargeException`

### Modificado: `modules/expo-media-session/src/index.ts`

- **Adicionado**: export `overrideNotificationArtwork(artworkUri: string)`

### Modificado: `src/services/radioService.ts`

- **Restaurado**: `setActiveForLockScreen(true, meta)` para titulo/artista + foreground service
- **Adicionado**: `ExpoMediaSession.overrideNotificationArtwork(meta.artworkUrl)` apos cada `setActiveForLockScreen` (so para URIs `file://`)
- **Restaurado**: `getLockScreenArtUri()` com file pool de 3 para unique paths

### Modificado: `src/utils/artworkCache.ts`

- **Restaurado**: `getLockScreenArtUri()` com `LOCK_POOL_SIZE = 3` — mantem 3 ficheiros em vez de deletar o anterior imediatamente
- **Mantido**: covers cache (`getLocalArtwork`, `getCachedArtwork`, `prefetchLogo`)

### Restaurado: `plugins/withStopAudioOnTaskRemoved.js`

Patch no AndroidManifest (NAO source code) — adiciona `android:stopWithTask="true"` ao `AudioControlsService`. Funciona porque patches de manifest afectam o merged manifest (ao contrario de patches Kotlin que nao afectam o AAR pre-compilado).

## Licoes aprendidas (v1.14.0 → v1.14.1)

1. expo-audio PRECISA de `setActiveForLockScreen(true)` para foreground service — sem isso, audio morre em ~3 min
2. `Notification.Builder.recoverBuilder()` permite reconstruir e re-postar uma notificacao existente com novos campos
3. O canal `expo_audio_channel` identifica a notificacao do expo-audio nos `activeNotifications`
4. Patches de manifest (withStopAudioOnTaskRemoved) FUNCIONAM — patches de source Kotlin NAO (AAR pre-compilado)
5. WiFi lock nao precisa de foreground service — pode ser adquirido directamente no modulo
