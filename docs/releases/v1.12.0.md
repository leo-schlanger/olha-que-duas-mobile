# Release v1.12.0 (Build 58)

**Data**: 2026-04-19
**versionName**: 1.12.0
**versionCode**: 58 (Android)

---

## Resumo da Release

Correcao definitiva da atualizacao de artwork e metadados (titulo, artista) ao mudar de musica em background. Aborda dois bugs independentes:

1. **In-app artwork (expo-image/Glide)**: O Glide pausa requests quando a Activity vai para background. Abordagens anteriores (`recyclingKey`, `key` simples) falhavam porque o React processava a mudanca de key enquanto a Activity ainda estava paused — o Glide enfileirava o request em vez de executar
2. **Lock screen notification (AudioControlsService)**: Quando duas musicas consecutivas usavam a mesma artworkUrl (fallback logo), `loadArtworkFromUrl` pulava o download E a reconstrucao da notificacao

---

## Causa Raiz — In-app artwork

expo-image usa **Glide** no Android. O `RequestManager` do Glide e lifecycle-aware:
- `onStop()` (background): pausa todos os requests
- `onStart()` (foreground): resume requests pendentes, mas NAO recarrega imagens ja carregadas

Quando o `key` do React muda em background (via subscription do nowPlayingService), React tenta unmount/remount a Image. Mas a nova `ExpoImageViewWrapper` recebe o mesmo `RequestManager` (singleton por Activity) que esta em estado paused. O request e enfileirado, nao executado. Quando a Activity resume, Glide resume requests pendentes, mas o componente ja esta marcado como "loaded".

### Fix: foregroundCount no key + reloadAsync()

```tsx
const [fgCount, setFgCount] = useState(0);
const imageRef = useRef<Image>(null);

useEffect(() => {
  const sub = AppState.addEventListener('change', (state) => {
    if (state === 'active') {
      setFgCount(c => c + 1);
      setTimeout(() => imageRef.current?.reloadAsync(), 200);
    }
  });
  return () => sub.remove();
}, []);

const imageKey = `${songKey}\0${fgCount}`;
<Image ref={imageRef} key={imageKey} ... />
```

O `foregroundCount` incrementa quando `AppState` muda para `'active'` (Activity ja resumida). O key muda APOS a Activity estar ativa → React unmount/remount acontece com o Glide em estado ativo → request executado imediatamente.

`reloadAsync()` via ref e chamado apos 200ms como seguranca adicional — forca `rerenderIfNeeded(force: true)` no lado nativo, contornando qualquer cache stale do Glide.

## Causa Raiz — Lock screen notification

Em `AudioControlsService.kt`, `loadArtworkFromUrl()` compara `url != currentArtworkUrl`. Se a URL e a mesma (e.g. fallback logo para duas musicas sem artwork cacheado), pula o download E o `postOrStartForegroundNotification()`. A notificacao nunca e reconstruida com os novos metadados.

### Fix: artworkUrl=undefined quando nao ha localArtUri

`pickArt()` agora retorna `data.localArtUri || undefined` em vez de `|| fallbackLogo`. Quando `artworkUrl` e null/undefined no Kotlin, o Elvis operator `?:` dispara `postOrStartForegroundNotification()` diretamente, reconstruindo a notificacao com os novos titulo/artista.

A primeira chamada (`setActiveForLockScreen`) continua a passar o logo URL para popular `currentArtwork` como bitmap de fallback.
