# Release v1.13.4 (Build 63)

**Data**: 2026-04-21
**versionName**: 1.13.4
**versionCode**: 63 (Android)

---

## Resumo da Release

Fix definitivo para lock screen artwork: solucao 100% JavaScript que
contorna o bug do expo-audio sem depender de patches nativos.

---

## A descoberta chave

O expo-audio (55.0.13) eh distribuido como **AAR pre-compilado** em
`node_modules/expo-audio/local-maven-repo/`. Os plugins Expo que patcham
o source Kotlin (`withMetadataFix.js`) NAO tem efeito porque o Gradle
usa o AAR (binario compilado), nao o source `.kt`.

Isso explica porque as v1.13.1-v1.13.3 nao resolveram o problema: os
patches estavam corretos no source mas nunca eram compilados.

## A solucao: unique file paths (100% JS)

Em vez de patchar o codigo nativo (impossivel com AAR pre-compilado),
a solucao contorna o bug da comparacao de URLs copiando o artwork para
um ficheiro com nome unico a cada troca de musica.

**Antes**: `file:///cache/covers/hash.jpg` ou `file:///logo.jpg#timestamp`
- `java.net.URL.equals()` ignora fragments → mesma URL → bitmap nao recarrega

**Depois**: `file:///cache/lockscreen/art-1713700000.jpg` (copia unica)
- URL genuinamente diferente a cada chamada → bitmap SEMPRE recarrega

### Ficheiros alterados

- `src/utils/artworkCache.ts` — nova funcao `getLockScreenArtUri()` que
  copia o artwork para `cache/olhaqueduas-lockscreen/art-{timestamp}.{ext}`,
  deleta o ficheiro anterior para evitar disk bloat
- `src/services/radioService.ts` — usa `getLockScreenArtUri()` em vez de
  fragments `#Date.now()` para todos os artworkUrl do lock screen
- Revertidos os hacks de debug (adService, GDPRConsent)

### Porque funciona

O `AudioControlsService.kt` (dentro do AAR) compara URLs com
`java.net.URL.equals()`:
- `file:///a.jpg#1` == `file:///a.jpg#2` (fragments ignorados) → SKIP
- `file:///art-001.jpg` != `file:///art-002.jpg` (paths diferentes) → LOAD

A copia de ficheiro (50-200KB) demora <5ms — imperceptivel.

## Licoes aprendidas

1. expo-audio eh distribuido como AAR pre-compilado, NAO compilado do source
2. Plugins que patcham source `.kt` em node_modules NAO afetam o build
3. `--no-build-cache --rerun-tasks` NAO ajuda — o AAR eh a dependency
4. Solucoes devem ser 100% JS quando o nativo nao pode ser patchado
