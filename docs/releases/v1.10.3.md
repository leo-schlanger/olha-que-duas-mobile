# Release v1.10.3 (Build 47)

**Data**: 2026-04-16
**versionName**: 1.10.3
**versionCode**: 47 (Android)

---

## Resumo da Release

Hot-fix focado no **bug raiz dos 9-16 segundos de delay** no lock screen.

A v1.10.2 implementou pre-cache de artwork mas tinha um bug: enquanto o
download não terminava, o `radioService` ainda passava o **URL HTTP da
música** ao native side como fallback. O `expo-audio` faz
`URL.openConnection().getInputStream()` **sem cache, sem timeout, sem
User-Agent** — em background, com cellular ou battery saver, esse fetch
pode bloquear 9-16 segundos e às vezes falhar silenciosamente.

Esta release garante que **o native side nunca recebe um URL remoto**: ou
recebe um `file://` (do artworkCache) ou recebe o logo pré-cacheado em
disco.

---

## Pesquisa que motivou a fix

Análise profunda de fontes 2026 ([github.com/expo/expo/issues/42900](https://github.com/expo/expo/issues/42900),
[@saltmango/expo-audio-stream](https://www.npmjs.com/package/@saltmango/expo-audio-stream),
[Akamai blog ExoPlayer](https://www.akamai.com/blog/performance/enhancing-video-streaming-quality-for-exoplayer-part-2-exoplayers-buffering-strategy-how-to-lower)):

- **expo-audio v1.1.1 (Expo SDK 54) não tem preloading** — buffering só
  começa após `play()`. PR #43061 adicionou preload mas só em v55.x.
- O caminho nativo `AudioControlsService.kt:loadArtworkFromUrl`:

  ```kotlin
  private fun loadArtworkFromUrl(url: URL, callback) {
    if (url != currentArtworkUrl) {
      currentArtworkUrl = url
      scope.launch {
        val inputStream = url.openConnection().getInputStream()  // ← lento
        val bitmap = BitmapFactory.decodeStream(inputStream)
        callback(bitmap)
      }
    }
  }
  ```

  Sem cache. Sem timeout. Sem User-Agent. Em background pode tardar 9-16s.
- A nossa v1.10.2 emitia primeiro com `localArtUri=null` + URL remoto da
  música, depois re-emitia com file:// quando download terminava. Mas o
  PRIMEIRO emit já tinha disparado o native fetch lento.

## Mudanças

### `src/utils/artworkCache.ts`

Novas funções:
- `prefetchLogo(remoteLogoUrl)` — chama `getLocalArtwork` para o logo e
  guarda o resultado num módulo-level `cachedLogoUri`.
- `getLogoUri(remoteLogoUrl)` — retorna síncronamente o `file://` cacheado
  ou cai no remoto (usado na PRIMEIRA chamada antes do prefetch terminar).

### `src/services/radioService.ts`

`subscribeToNowPlaying`: o callback **nunca** mais passa um URL remoto da
música ao `updateLockScreenMetadata`:

```typescript
// ANTES (v1.10.2):
const pickArt = (remote) => data.localArtUri || remote || siteConfig.radio.logoUrl;
//                                                ↑
//                          passava a URL HTTP da música
//                          → native fetch bloqueava 9-16s em bg

// DEPOIS (v1.10.3):
const fallbackArt = getLogoUri(siteConfig.radio.logoUrl);  // file:// cacheado
const pickArt = () => data.localArtUri || fallbackArt;
//                                          ↑
//                          sempre file:// (cache art OU logo cacheado)
//                          → native load <10ms, fiável em bg
```

Aplicado em `music`, `podcast`, `announcement`, `liveShow` e idle.

A primeira chamada a `updateLockScreen` em `play()` (que dispara
`setActiveForLockScreen`) também usa `getLogoUri` em vez do URL remoto.

### `App.tsx`

Adicionado `prefetchLogo(siteConfig.radio.logoUrl)` ao `initializeServices`,
fire-and-forget. Quando a app está pronta, `cachedLogoUri` já está
populado.

## Resultado esperado

**Lock screen image**:
- Cache hit (música tocada antes nesta sessão): instantâneo, file:// disk
- Cache miss (música nova): ~500ms-1s para mostrar a arte (download em
  paralelo com emit; quando termina, re-emit com file://)
- Fallback (no localArtUri): logo cacheado, file:// disk, instantâneo
- **Nunca** mais bloqueia em fetch remoto na thread nativa

**Audio startup**: continua limitado pelo prebuffer do ExoPlayer (~2-3s).
expo-audio 1.1.1 não expõe config. Fix real precisa de Expo SDK 55 (que
inclui expo-audio 55.x com PR #43061 de preload) OU migração para
react-native-track-player.

---

## Limitações conhecidas

Esta release fixa o caminho do **lock screen artwork**. Se ainda houver
delay perceptível após v1.10.3, são separadas:

| Sintoma | Causa | Solução |
|---------|-------|---------|
| Lock screen art demora 9-16s | URL remoto ao native | **FIXADO em v1.10.3** |
| Audio startup demora 2-4s no tap play | ExoPlayer prebuffer (não configurável em expo-audio 1.x) | **Requer migração**: Expo SDK 54→55 (expo-audio com preload) ou react-native-track-player |
| Áudio para após ~3min em background | [expo-audio bug #38317](https://github.com/expo/expo/issues/38317) | Fix do upstream (PR #38980), provavelmente já em 1.1.1; se persistir, requer SDK 55 ou RNTP |
| Throttling de algumas OEMs em background (Xiaomi/Huawei) | OS-level battery optimization | Não-resolvível por código de app; utilizador tem que permitir auto-start |
