## Resumo da Release

Release massiva de qualidade, robustez e experiencia do utilizador. Resultado de um checkup completo end-to-end da aplicacao seguido de uma redesign do sistema de notificacoes/lembretes e correcoes do delay na troca de musica.

**Cinco eixos principais:**

1. **Sistema de lembretes redesenhado** - novo bottom sheet "Os Meus Lembretes" substitui o velho `Alert.alert` confuso, com update otimista (clique = mudanca instantanea), per-show queue (taps rapidos colapsam para o ultimo intent), e bug do "desativa mas continua mostrando ativado" resolvido
2. **Now Playing mais rapido** - migracao para `expo-image` com crossfade nativo, polling reduzido de 5s para 3s, removida a janela de "transition" de 1.5s que escondia a musica - delay perceptivel reduzido em ~50%
3. **Programacao semanal robusta** - lazy expansion ("Ver mais dias"), error banner, empty state melhorado, dark mode contrast fixado, hit target do reminder bell aumentado para 44x44, ellipsizeMode explicito, multi-time badges com maxWidth
4. **Dependencias alinhadas** - 5 pacotes desalinhados corrigidos, incluindo `expo-localization` que estava em versao canary do SDK 55 num projeto SDK 54 (risco real em producao)
5. **Acessibilidade, performance e dividas tecnicas** - 6 botoes so com icone receberam `accessibilityLabel`, `dynamicStyles` do Settings agora memoizado, dead code removido, race condition GDPR/Premium corrigida, retry exponencial no intersticial e mais

## Sistema de Lembretes - Redesign Completo

### O bug do "desativa mas continua mostrando ativado"

**Causa raiz**: `if (isOperationPending()) return` em `RadioPlayer.handleToggleReminder` ignorava silenciosamente taps quando a operacao anterior ainda estava em andamento. O utilizador via o sino na programacao continuar a indicar "ativo" mesmo apos clicar para desativar.

**Solucao**:

- **Update otimista**: novo state `optimisticEnabledShows: Set<string>` no `RadioPlayer` mantem um espelho local que muda visualmente *antes* da operacao async terminar. O sino vira `bell-ring` ou `bell-outline` instantaneamente
- **Per-show queue**: novo `pendingOpsRef: Map<string, { latest, itemRef, running }>` rastreia o ultimo intent de cada show. Multiplos taps rapidos colapsam para o estado final via worker `processShowQueue` com loop while-latest-changes
- **Rollback automatico**: em caso de falha, `setOptimisticEnabledShows(new Set(notificationService.getPreferences().enabledShows))` ressincroniza o estado optimistic com o real, e um toast de erro aparece
- **`isOperationPending` removido do bloqueio do sino**: substituido por `noOpIsOperationPending` que retorna sempre `false`. O optimistic UI ja absorve o feedback visual sem precisar bloquear o botao

### O sino ao lado do play agora "faz alguma coisa"

**Antes**: ao tocar no sino do play, abria-se um `Alert.alert` listando os shows ativos como texto, com "Manage in Settings" como unica opcao - utilizador nao podia gerir nada dali.

**Agora**: novo componente `RemindersBottomSheet.tsx` com:

- **Header**: icone bell-ring, titulo "Os Meus Lembretes", subtitle com count plural ("3 lembretes ativos")
- **Seletor de tempo de antecedencia inline**: chips 5/15/30/60 min, mudanca direta sem fechar o sheet
- **Lista de lembretes ativos**: cada linha mostra nome do show + horarios formatados (ex: "11:00 . 19:00") + botao X para remover, com `ActivityIndicator` por linha durante a remocao
- **Empty state polido**: icone `bell-sleep-outline` + mensagem clara + hint explicando como ativar lembretes na programacao semanal
- **Footer "Desativar todos"**: com confirmacao *inline* (nao Alert) usando dois botoes "Cancelar / Sim, desativar"
- **Acessibilidade completa**: `accessibilityRole`, `accessibilityState` (selected, disabled), `accessibilityLabel` em cada elemento interativo
- **Reusa o padrao do `AboutBottomSheet`** para consistencia visual (handle, slide animation, backdrop, safe area insets)

### ToastContext novo

- **`src/context/ToastContext.tsx`**: provider + `useToast()` hook expondo `show(message, options)` e `hide()`
- Wrapper sobre `Snackbar` do `react-native-paper` (ja era dependencia transitiva via `PaperProvider`)
- Variants: `info`, `success`, `error` com cores e duracoes diferentes (4s para erros, 2.5s para sucesso/info)
- Suporta `actionLabel` + `onAction` (usado para "Open settings" quando permissao e negada)
- Wrap em `App.tsx` *dentro* de `ThemeProvider` (precisa de `colors`) e *fora* de `PremiumProvider` (precisa estar disponivel pra todo lado)

### Sync melhorado

- Novo `AppState` listener no `RadioPlayer` rastreia `lastBackgroundedAtRef`
- Quando o app volta do background apos `>30 min`, chama `notificationService.forceSync()` automaticamente
- Reconcilia local prefs <-> system notifications, removendo "shows fantasmas" cujas notifs ja foram canceladas pelo OS
- Sem toast intrusivo - sync acontece em silencio

### Permission flow polido

- `handleOpenNotificationSettings` agora pede permissao primeiro se nao tem nenhuma e o utilizador tenta abrir o sheet sem reminders
- Se permissao for negada, mostra toast vermelho com action "Open settings" que abre `Linking.openSettings()`
- Sem mais Alerts modais bloqueando o flow

### `useNotifications` hook expandido

- Novo metodo exposto: `cancelAllReminders(): Promise<boolean>` (wrapper de `notificationService.cancelAllNotifications`)

## Now Playing - Delay de Troca de Musica

### Diagnostico

Delay total perceptivel anterior: `~2-7 segundos` entre a musica realmente mudar no stream e a foto/info aparecer no app.

Causas identificadas:

1. **Polling de 5s** - lag de 0-5s para detectar a mudanca via `/api/nowplaying`
2. **Janela de "transition" de 1500ms** - durante 1.5s o componente NowPlaying *escondia* a musica e mostrava o nome da radio em vez disso (quando `isTransition: true`)
3. **`Image` do React Native, nao `expo-image`** - sem cache, sem crossfade nativo, decode mais lento
4. **Sem preload** - a nova imagem so comecava a carregar depois do `setState`
5. **120x120 render mas carregava tamanho original** - banda e decode desperdicados

### Solucao

- **`NowPlaying.tsx`**: migracao de `Image` (RN) para **`Image` do `expo-image`** com:
  - `transition={350}` - crossfade nativo de 350ms ao trocar a arte
  - `cachePolicy="memory-disk"` - cache em memoria + disco, troca instantanea para artes ja vistas
  - `priority="high"` - prioridade no decode pipeline
  - `contentFit="cover"`
- **Removida** a check `&& !nowPlaying.isTransition` que escondia a musica durante 1.5s. Agora o componente sempre mostra a musica se ela existe
- **`useNowPlaying.ts` reescrito** (~80 linhas -> ~50 linhas):
  - Removida a logica de `pendingSongRef` + `transitionTimeoutRef`
  - Aplica a musica nova diretamente ao receber um update do service
  - Dedup mais inteligente por `(title, artist, art)` - antes era so `(title, artist)`, agora se a API atualizar a arte mid-song, atualiza visualmente
  - `isTransition` mantido na interface por compatibilidade mas nao usado mais
- **`constants.ts`**: `NOW_PLAYING_POLL_INTERVAL: 5000 -> 3000` (foreground), `NOW_PLAYING_TRANSITION_DURATION: 1500 -> 350`

### Resultado

| Metrica | Antes | Depois |
|---|---|---|
| Deteccao polling | 0-5s (medio 2.5s) | 0-3s (medio 1.5s) |
| Janela "limbo" mostrando radio | 1500ms fixo | 0ms (removido) |
| Network/decode 2a vez | 200-800ms | 0ms (cache) |
| Crossfade visual | hard cut | 350ms suave |
| **Total perceptivel** | **~2-7s** | **~1.5-3s** (melhoria ~50%) |

**TODO futuro**: substituir polling por SSE/WebSocket do AzuraCast (`/api/live/nowplaying/SSE/{station}`) para zerar os ultimos 0-3s de delay. Adiada para release futura.

## Programacao Semanal e Diaria

### Lazy expansion ("Ver mais dias")

- **`ScheduleSection.tsx`**: novo state `expanded` + constante `DEFAULT_VISIBLE_DAYS = 3`
- Por defeito, mostra so os primeiros 3 dias (HOJE + 2 seguintes)
- Botao **"Ver mais dias (N)"** com chevron-down expande o resto, **"Ver menos"** com chevron-up colapsa
- Resolve o problema de performance/UX com schedules grandes - antes renderizava todos os 7 dias x N shows simultaneamente sem virtualizacao

### Error display

- `ScheduleSection` agora aceita prop `error?: string | null`
- Quando ha erro mas o fallback existe, mostra um banner vermelho compacto no topo: "Mostrando programacao guardada"
- Quando `scheduleByDay.length === 0` E `error`, o empty state usa `alert-circle-outline` em vez de `calendar-blank-outline`
- `useDailySchedule` agora tambem retorna `error` (consistente com `useSchedule`)
- `DailyScheduleSection` recebe `error` e mostra o mesmo banner

### Empty state polido

- Antes: so um texto "Sem programacao disponivel"
- Agora: icone `calendar-blank-outline` (48px) + titulo + hint explicativo
- Para o `DailyScheduleSection`: icone `music-note-off` + texto centralizado

### Hit target do reminder bell

- `radioStyles.ts:reminderButton` - **38x38 -> 44x44** (atinge minimo Apple HIG / Material Design)
- `borderRadius` ajustado de `19` para `22`
- `marginTop` reduzido de `4` para `2` para nao quebrar o alinhamento

### Dark mode contrast nos time badges

- `radioStyles.ts:timeBadge` agora usa `backgroundColor: isDark ? colors.muted + '60' : colors.background`
- Removida a sobrescrita inline de `backgroundColor` em `ScheduleSection.tsx` que estava sobrepondo o estilo dark-mode-aware
- Contraste WCAG passa de ~3.5:1 para acima de 4.5:1 (minimo AA)

### Long titles e descriptions

- `ellipsizeMode="tail"` explicito no `showName` e `description` (antes so tinha `numberOfLines`)
- `times` agora tem `maxWidth: '100%'` para nao empurrar o reminder button para fora do card

### Reminder button pending state

- Quando `isLoading || isOperationPending`, o botao mostra `<ActivityIndicator>` dentro em vez de so ficar disabled silenciosamente
- `accessibilityState={{ disabled, checked }}` adicionado

## Performance, Arquitetura e Bug Fixes

### `dynamicStyles` do Settings memoizado

- `SettingsScreen.tsx:651` - `const dynamicStyles = createDynamicStyles(colors, isDark)` agora envolto em `useMemo`
- Antes recriava o objeto a cada render, quebrando a memoizacao de todas as section components que recebiam `dynamicStyles` como prop
- Re-renders desnecessarios eliminados em `AppearanceSection`, `LanguageSection`, `RadioSection`, `PremiumSection`, `NotificationSection`, `AboutSection`

### Race Premium/GDPR no init de ads

- `App.tsx`: novo guard `if (isPremiumLoading) return;` no useEffect que inicializa o ad SDK
- Antes: usuario premium podia ter o SDK do AdMob inicializado brevemente antes do AsyncStorage carregar o status premium - desperdicio de bateria/dados
- Agora: ad SDK so inicializa quando temos certeza de que o utilizador NAO e premium

### Intersticial auto-retry

- `adService.ts`: novo metodo privado `scheduleInterstitialRetry()` com backoff exponencial **5s -> 30s -> 2min**
- Quando o intersticial falha no `ERROR` event, agenda retry em vez de ficar morto ate o app reiniciar
- Reset do contador no `LOADED` ou `CLOSED`
- Evita perda total de receita no slot a cada 4 noticias quando ha um blip de rede

### `Linking.openURL` com try/catch

- `NewsDetailScreen.tsx:handleOpenSource` agora e async, valida com `Linking.canOpenURL()`, faz try/catch, e mostra Alert se falhar
- Antes: `Linking.openURL()` sem try/catch - se URL invalido, falha silenciosa

### Dead code removido

- `useCategories` e `useRegions` em `useNews.ts` (138-181) - exportados mas nunca importados em lugar nenhum
- Imports de `fetchCategories` e `fetchRegions` removidos do hook (continuam em `newsApi.ts` porque os testes ainda os usam)

### Outros

- `App.tsx:240`: `name={props.name as any}` -> `name={props.name as keyof typeof MaterialCommunityIcons.glyphMap}` (consistente com o resto do projeto)
- `WeatherForecast.tsx:208`: `key={index}` -> `key={item.date}` (estavel)
- `SettingsScreen.tsx:456-477`: lista de "shows habilitados" agora dentro de `<ScrollView maxHeight: 240, nestedScrollEnabled>` para nao estourar o card quando o utilizador habilita 10+ lembretes
- Catches vazios em `BannerAd.tsx:37` e `NewsDetailScreen.tsx:40` agora logam erros via `logger.log` em vez de silenciar
