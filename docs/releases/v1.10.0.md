1. **Performance da rádio + background** — corrige engasgo, fotos não atualizam em background, e radio que parava ao premir Home

## Performance da Rádio

### Visualizer (eliminação do engasgo)

`src/components/radio/RadioVisualizer.tsx`

| Métrica | Antes | Depois |
|---|---|---|
| Barras | 12 | 7 |
| Intervalo de tick | 150ms | 220ms |
| Animações/segundo | ~80 | ~32 |
| Em background | continuava a animar | pausa via `AppState` |

Cada `Animated.timing` faz roundtrip JS↔native bridge mesmo com `useNativeDriver` (o native driver só assume depois de o JS agendar). 80 anims/seg em low-end Android era a maior fonte de pressão na bridge e contribuía para microcortes no áudio.

### Fotos / título no lock screen em background

`src/services/nowPlayingService.ts`

**Antes:** ao ir para background, fechava o SSE e caía para polling de **15s**. Resultado: capa do álbum no lock screen demorava até 15s a refletir uma troca de música.

**Depois:** mantém o SSE aberto em background. O foreground service do `expo-audio` mantém o JS context vivo enquanto a rádio toca, logo a socket sobrevive. Polling de fallback acelerado de **15s → 6s** (caso o SSE caia mesmo assim). Resultado: actualização em segundos em vez de minutos.

### Rádio para de tocar ao premir Home (background por defeito)

`src/services/radioSettingsService.ts`

Default `stopOnClose: true → false`. Era contraditório com `backgroundPlayback: true` e estava a parar a rádio sempre que o utilizador premia Home ou apagava o ecrã. O plugin nativo `withStopAudioOnTaskRemoved` continua a parar correctamente quando o utilizador faz swipe-to-kill.

Utilizadores existentes mantêm a sua configuração (`load()` faz merge com defaults).

## Botão "Manter Ecrã Ligado"

Pacote novo: `expo-keep-awake@~15.0.8`

### Reorganização dos botões

**Ao lado do play** (antes: sino que abria gestão de lembretes):

- Botão **lightbulb** (`lightbulb-on` / `lightbulb-outline`)
- Toggle: quando activado, o ecrã não dorme enquanto estás na rádio (útil em cozinha, suporte de carro)
- Wake lock libertado automaticamente ao desactivar o toggle E ao desmontar o componente (sem leak se navegares com keep-awake on)
- Tag única `'olhaqueduas-radio'` para `activateKeepAwakeAsync` — evita colisão com outros usos

**No cabeçalho da grelha semanal** (novo):

- Botão **sino** com badge da contagem de programas com lembrete activo
- Abre o `RemindersBottomSheet` igual ao anterior (gestão global: tempo de aviso, desactivar todos)

**Em cada programa da grelha**: o sino individual continua exactamente como estava.

### Implementação

`src/components/RadioPlayer.tsx`:
- Estado `keepAwake` + `handleToggleKeepAwake` com `activateKeepAwakeAsync` / `deactivateKeepAwake`
- Cleanup no unmount: `deactivateKeepAwake(KEEP_AWAKE_TAG)` em `useEffect` return

`src/components/radio/RadioControls.tsx`:
- Props: `hasActiveNotifications`/`notificationCount`/`onNotificationPress` removidas
- Props: `keepAwake`/`onToggleKeepAwake` adicionadas
- `accessibilityRole="switch"` + `accessibilityState={{ checked: keepAwake }}`

`src/components/radio/ScheduleSection.tsx`:
- Props novas: `activeReminderCount` + `onOpenReminders`
- Botão sino no header com badge

### i18n

- `pt.json` / `en.json`: `radio.controls.keepAwakeOn` / `keepAwakeOff`

## Checkup Geral — Fixes

### [ALTA] Init concorrente em `App.tsx`

`initInFlightRef` previne duplicações se o utilizador clicar Retry duas vezes (ou React StrictMode em dev disparar duas vezes). Também limpa em `finally`.

### [ALTA] IAP: compras tardias após timeout

`src/services/purchaseService.ts` + `src/context/PremiumContext.tsx`

Adicionado novo método `onPurchaseDetected(listener)` no purchaseService — observers de longa duração que disparam para QUALQUER compra válida, incluindo as que chegam depois do timeout de 2 minutos. `PremiumContext` regista um listener no mount e marca `isPremium = true` automaticamente — sem necessidade de o utilizador reiniciar o app.

### [MED] SSE close race

`src/services/nowPlayingService.ts:closeSSE`

Detacha `eventSource = null` **síncrono** antes de `removeAllEventListeners()` + `close()`. Handlers em flight que disparem entre as duas chamadas vêem `eventSource === null` e short-circuit (`handleSSEMessage` / `handleSSEDisconnect` já fazem branch nisso).

### [MED] adService: leak de listeners em load failure

`src/services/adService.ts:loadInterstitial`

Trackeamos unsubs incrementalmente num array local `unsubsSoFar`. Se uma das chamadas `addAdEventListener` falhar a meio, o `catch` invoca os unsubs já recolhidos antes de zerar o `interstitialAd`.

### [MED] News: erros diferenciados

`src/hooks/useNews.ts` + `src/services/newsApi.ts`

Novo `mapNewsErrorToKey(err, fallback)` distingue:
- **offline** — `TypeError: Network request failed` / `failed to fetch`
- **timeout** — `AbortError` / mensagem com 'timeout'
- **notFound** — Supabase `PGRST116` (single row not found) / HTTP 404
- **server** — HTTP 5xx
- **fallback** — chave default

`fetchNewsById` agora distingue "slug genuinamente inexistente" (retorna `null` → screen mostra `news.notFound`) de "erro de rede/servidor" (lança → screen mostra erro específico).

i18n: `news.errors.{offline,timeout,notFound,server}` + `news.notFound` em PT e EN.

### [BAIXA] Permission denial permanente em `useLocation`

`src/hooks/useLocation.ts` + `src/screens/WeatherScreen.tsx`

Novo estado `'denied-permanent'` quando o utilizador escolheu "Don't ask again". `WeatherScreen.handleRetry` agora:
- `denied-permanent` → abre directamente Settings (botão diz "Abrir definições")
- `denied` → tenta requestPermission de novo
- outros → refresh

### [BAIXA] `NewsDetailScreen` `t(error)` defensivo

Passa `defaultValue: t('news.detailError')` para que strings raw que escapem o mapeamento não fiquem visíveis cruas ao utilizador.
