## Pre-download de Artwork (`src/utils/artworkCache.ts`)

### O problema residual da v1.10.1

A v1.10.1 corrigiu o caminho — `updateLockScreenMetadata` em vez de
`setActiveForLockScreen` — para que o lock screen actualizasse fiavelmente em
background. Mas o caminho nativo do `expo-audio`
(`AudioControlsService.kt:301-314`) ainda fazia download fresh da imagem por
cada música:

```kotlin
private fun loadArtworkFromUrl(url: URL, callback: (Bitmap?) -> Unit) {
  if (url != currentArtworkUrl) {
    currentArtworkUrl = url
    scope.launch {
      val inputStream = url.openConnection().getInputStream()  // <-- 200ms-2s
      val bitmap = BitmapFactory.decodeStream(inputStream)
      callback(bitmap)
    }
  }
}
```

Sem cache, sem retry, sem timeout. Em background com cellular ou battery
saver, este fetch pode falhar silenciosamente — utilizador vê a capa antiga
(ou nenhuma).

### Solução

Novo módulo `src/utils/artworkCache.ts` que pré-baixa a imagem para um
ficheiro local **antes** de chegar ao native side:

```typescript
export async function getLocalArtwork(remoteUrl: string): Promise<string | null> {
  // FNV-1a hash da URL → filename estável
  const file = new File(coversDir, `${hashUrl(remoteUrl)}.jpg`);

  if (file.exists) return file.uri;  // <-- cache hit, instantâneo

  // Download via expo-file-system (modern API)
  const downloaded = await File.downloadFileAsync(remoteUrl, file);
  return downloaded.uri;
}
```

**Cache details:**

- Pasta: `Paths.cache/olhaqueduas-covers/` (limpa pelo SO em low storage; OK)
- Filename: `{FNV-1a 32-bit hash da URL}.{jpg|png|webp|gif}` — determinístico
- Limite: **25 ficheiros** (purga oldest a cada 10 downloads, ordenado por mtime)
- Coalescing: chamadas concorrentes para a mesma URL partilham um `Promise`
- Cache hit síncrono via `getCachedArtwork(url)` para emit imediato

### Fluxo completo de uma troca de música

```
1. SSE delivery        ─→ nowPlayingService.processPayload()
2. emit():
   - getCachedArtwork(art) síncrono
     → cache hit?  → data.localArtUri = file://...
                     emit aos listeners (lock screen instant!)
     → cache miss? → emit aos listeners com remote URL
                     getLocalArtwork(art) async em paralelo
                     ┌──────────────────────────────────────┐
                     │ Download em ~200-800ms               │
                     │ Quando termina:                      │
                     │   if (current song ainda é a mesma): │
                     │     re-emit com data.localArtUri     │
                     │     → lock screen swap instantâneo   │
                     └──────────────────────────────────────┘
3. radioService subscriber:
   - data.localArtUri ?? data.song.art ?? logoUrl
   - updateLockScreenMetadata({ artworkUrl })
4. expo-audio nativo:
   - Se file://: BitmapFactory.decodeStream do disco em <10ms
   - Se http://: openConnection() download como antes (fallback)
```

### Mudanças nos types

`NowPlayingData` ganha `localArtUri: string | null`.

`dataChanged()` agora considera `localArtUri` para que o re-emit pós-download
não seja descartado pela deduplicação.

`useNowPlaying.IDLE` actualizado.

## Strings Actualizadas

### `settings.radio.stopOnCloseDesc`

- **PT antigo**: "A rádio para quando fechar a aplicação"
- **PT novo**: "Por defeito a rádio continua a tocar em background. Ative para parar quando fechar a app."
- **EN antigo**: "Radio stops when closing the app"
- **EN novo**: "By default the radio keeps playing in the background. Enable to stop when closing the app."

**Razão**: o default de `stopOnClose` mudou para `false` em v1.10.0, o
texto continuava a sugerir o comportamento antigo.

### `notifications.noActiveReminders.message`

- **PT antigo**: "...sininho ao lado de cada programa na lista abaixo."
- **PT novo**: "Toque no sino ao lado de qualquer programa na grelha semanal para receber um aviso antes de começar."
- **EN**: equivalente

**Razão**: "lista abaixo" referia o sino do RadioControls (removido). Agora
o sino vive na grelha semanal.

### `notifications.management.emptyHint`

Reescrito em PT/EN para mencionar **ambos** os pontos de entrada (sino do
header + sino por programa).

## Como Validar

1. **Tap play** → tocar 1 música até ao fim
2. **Bloquear ecrã** → esperar 1-2 trocas de música
3. **Desbloquear** → notificação no lock screen deve mostrar **a foto E o título da música ACTUAL**, não a antiga
4. **Tocar a mesma música pela 2ª vez** (forçar via web admin se preciso): foto deve aparecer **instantaneamente** na lock screen (cache hit)
5. **Pressionar Home** durante reprodução, esperar 30s, voltar à app: rádio continua, foto na notificação deve estar correcta

Se algum dos passos falhar com a v1.10.2, partilha o sintoma exacto.
