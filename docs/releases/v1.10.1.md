# Release v1.10.1 (Build 45)

**Data**: 2026-04-16
**versionName**: 1.10.1
**versionCode**: 45 (Android)

---

## Resumo da Release

Patch focado em **rádio fluida e foto em background** — investigação agressiva à pipeline completa entre tap e som limpo.

Cinco mudanças críticas:

1. **Lock screen agora actualiza arte/título em background** — descoberta a causa raiz: `setActiveForLockScreen()` recriava a `MediaSession` inteira a cada música, podendo até reiniciar o foreground service. Trocas de música agora usam `updateLockScreenMetadata()` (caminho leve do expo-audio) que só refresca a notificação.
2. **Resume instantâneo após pause** — `play()` reusa o player existente (~100ms) em vez de o destruir e recriar (~1-3s).
3. **Image prefetch ao receber metadata** — artwork entra na cache de `expo-image` antes do componente renderizar; sem flash branco.
4. **Visualizer 100% native-driven** — substituído `setInterval`+`Animated.timing` por `Animated.loop` com `interpolate`. Zero JS work após setup.
5. **Init de ads diferido 4s** — não compete por CPU/network com o startup do stream.

## Lock Screen — fix da foto em background

### A causa real

Análise do código nativo do `expo-audio@1.1.1` (`AudioControlsService.kt:200-255`) revelou:

```
setActivePlayerInternal(player, metadata):
  remove playback listener
  hide notification
  release MediaSession              // <-- destrói
  create NEW MediaSession           // <-- recria
  loadArtworkFromUrl                // <-- pode falhar em bg
  postOrStartForegroundNotification(startInForeground = true)  // <-- restart!
```

**Cada chamada a `setActiveForLockScreen(true, meta)` reinicia o foreground service de mídia.** No Android isto:
- Pode haver um pequeno gap onde a notificação desaparece e re-aparece
- Em background com algumas OEMs (Xiaomi, Huawei), o restart pode ser atrasado pelo sistema
- A nova `MediaSession` precisa de re-fetch da arte; se a network estiver throttled em bg, fica sem imagem

### O caminho leve

`updateLockScreenMetadata(meta)` chama `updateMetadataInternal()`:

```
updateMetadataInternal(player, metadata):
  if metadata == currentMetadata: return
  currentMetadata = metadata
  loadArtworkFromUrl                                              // mesma carga de imagem
  postOrStartForegroundNotification(startInForeground = false)    // só refresca!
```

Sem destruir a session, sem reiniciar o foreground service. Funciona fiável em background.

### Implementação

`src/services/radioService.ts`:

- Novo flag `lockScreenSessionActive: boolean`
- `updateLockScreen(meta)`:
  - Primeira chamada por player (`!lockScreenSessionActive`) → `setActiveForLockScreen(true, meta)` + flag = true
  - Chamadas seguintes → `updateLockScreenMetadata(meta)`
- `resetLockScreenCache()` reset do flag em `play()` (slow path) / `stop()` / `cleanup()`
- `pause()` NÃO reseta o flag — a session fica viva durante o pause para resume rápido

## Resume instantâneo (player reuse)

`src/services/radioService.ts:play()`

Antes:
```
play():
  release old player        // ~100ms
  create new player         // ~50ms
  TCP handshake             // ~200ms
  ICY headers + prebuffer   // ~500-2500ms (ExoPlayer DEFAULT_BUFFER_FOR_PLAYBACK_MS)
  play()
  ----- até som: 1-3s -----
```

Agora (fast path em pause→play):
```
play():
  if (player exists):
    player.play()           // <100ms (connection ainda viva)
    ----- até som: ~100ms -----
```

Slow path (primeiro play / recreate após erro) continua o caminho antigo.

`pause()` mantém o player vivo (não chama `release()`) e mantém a `MediaSession` activa para o utilizador poder retomar do lock screen.

## Image prefetch

`src/services/nowPlayingService.ts:emit()`

```typescript
let artUrl: string | null = null;
if (data.mode === 'music' && data.song?.art) artUrl = data.song.art;
else if (data.mode === 'podcast' && data.podcastArt) artUrl = data.podcastArt;
else if (data.mode === 'announcement' && data.announcementArt) artUrl = data.announcementArt;
if (artUrl) {
  ExpoImage.prefetch(artUrl).catch(() => { /* best effort */ });
}
```

`expo-image` faz HEAD + GET assíncronos e popula a cache `memory-disk`. Quando `<NowPlaying>` re-renderiza com a nova URL e usa `<Image cachePolicy="memory-disk">`, a imagem já está disponível localmente — render instantâneo, sem flash do fallback.

## Visualizer — Animated.loop nativo

`src/components/radio/RadioVisualizer.tsx`

### Antes (v1.10.0)

7 barras × `setInterval(220ms)` × `Animated.timing` por barra = **~32 calls/segundo no JS thread**, cada uma cruza a bridge JS↔native (mesmo com `useNativeDriver`, o native driver só assume DEPOIS do JS agendar).

### Agora

```typescript
const driver = new Animated.Value(0);
Animated.loop(
  Animated.timing(driver, { toValue: 1, duration: 1500, useNativeDriver: true })
).start();

// Cada barra:
const scale = driver.interpolate({
  inputRange: [...keyframes com phase shift...],
  outputRange: [...padrão pseudo-random fixo por barra...],
});
```

Zero JS work após o `start()` inicial. O native driver corre o loop 100% no thread nativo, computa as interpolações natively, e renderiza a 60fps sem nunca tocar no JS.

Padrões pseudo-random gerados uma vez à carga do módulo (LCG seedeado) — visualmente parece "vivo" e variado, mas é determinístico e barato.

Pausa em background mantida (via AppState).

## Diferimento da init de ads

`App.tsx`

Antes: `adService.initialize` + `loadInterstitial` corriam imediatamente após `adsConsent` resolver, em paralelo com o stream startup.

Agora: `setTimeout(...)` de **4s**. O stream tem tempo de estabelecer a connection, prebuffer e estabilizar antes de o AdMob SDK fazer os seus pedidos de inicialização.

## Loggers em hot paths

`src/services/radioService.ts`

Removidos `logger.log` em:
- `pollPlayerStatus` no branch de "status mudou" (disparava a cada mudança a 1.5s)
- `handlePlaybackStatus` no branch de "status mudou" (até 2x/segundo a default 500ms)

Mantidos: `logger.error` e `logger.warn` (raros, importantes).

Embora `logger.*` seja gated por `__DEV__`, a função call ainda tem custo. E em dev mode, a saída de console num device físico via Metro pode causar lag visível.

## Ordem de impacto perceptível

1. **Foto em background** (lock screen): impacto IMEDIATO — agora usa o caminho que o expo-audio desenhou para isso.
2. **Resume após pause**: visível ao utilizador — feedback "play" → som em ~100ms vs ~1-3s.
3. **Visualizer**: invisível no display, mas elimina pressão constante na bridge durante playback. Devices low-end deverão sentir reprodução mais suave.
4. **Image prefetch**: elimina o "flash" da capa nova quando a música muda na pantalha.
5. **Ads diferidos**: melhoria do startup latency da rádio na primeira sessão.
