# Release v1.16.0 (Build 67)

**Data**: 2026-04-27
**versionName**: 1.16.0
**versionCode**: 67 (Android)

---

## Resumo da Release

Reescrita completa da arquitetura de MediaSession e notificacao de midia. O expo-audio deixa de gerir a notificacao, lock screen e MediaSession. O modulo nativo `ExpoMediaSession` (Kotlin) passa a ser dono de tudo, eliminando todas as race conditions de artwork e a necessidade de patches no expo-audio.

---

## Problema

Desde a v1.11.4, o lock screen artwork foi fonte constante de bugs:

- **v1.13.1-v1.13.4**: Multiplas tentativas falhadas (fragment cache-buster, patches nativos, file pools)
- **v1.14.0**: Tentativa de servico proprio falhou porque expo-audio precisa de `setActiveForLockScreen` para foreground service
- **v1.14.1**: Override hibrido com race conditions (expo-audio sobrescrevia artwork)
- **v1.15.0**: Solucao "race-free" funcional mas fragil: dependia de `recoverBuilder()`, 6 retries com delays, e patch no expo-audio

A cada fix, novos edge cases surgiam. O expo-audio patch precisava ser mantido a cada atualizacao de SDK.

---

## Solucao: Arquitetura propria de MediaSession

### Separacao clara de responsabilidades

| Componente | Antes (v1.15.0) | Depois (v1.16.0) |
|---|---|---|
| Foreground Service | expo-audio (AudioControlsService) | Nosso MediaService |
| MediaSession | expo-audio (via setActiveForLockScreen) | Nosso MediaService (android.media.session) |
| Notificacao | expo-audio cria + modulo "remenda" artwork | MediaService cria e gerencia tudo |
| WiFi Lock | Duplicado (patch + modulo) | Unico, no MediaService |
| Artwork | Dual-track + 6 retries com delays | updateMetadata direto |
| Lock screen controls | Polling a cada 5s detectava mudancas | Eventos nativos (onRemotePlay/Pause/Stop) |

### Ficheiros criados

- **`MediaService.kt`** ‚Äî Foreground service Android com:
  - `android.media.session.MediaSession` (lock screen, botoes de midia, Bluetooth)
  - Notificacao com artwork, titulo, artista, botoes play/pause/stop
  - WiFi lock (previne WiFi off com ecra desligado)
  - Bitmap loading em thread dedicada (scale 512x512, opaque background)

### Ficheiros reescritos

- **`ExpoMediaSessionModule.kt`** ‚Äî Nova API nativa:
  - `activate(title, artist, artworkUri)` ‚Üí inicia servico foreground
  - `updateMetadata(title, artist, artworkUri)` ‚Üí atualiza notificacao
  - `updatePlaybackState(isPlaying)` ‚Üí atualiza estado play/pause
  - `deactivate()` ‚Üí para servico, remove notificacao
  - Eventos: `onRemotePlay`, `onRemotePause`, `onRemoteStop`

- **`index.ts`** + **`ExpoMediaSessionModule.ts`** ‚Äî API TypeScript tipada com suporte a eventos

### Ficheiros modificados

- **`radioService.ts`** ‚Äî Removidos todos os `setActiveForLockScreen`, dual-track artwork, WiFi lock manual. Adicionados listeners para eventos remotos.
- **`AndroidManifest.xml`** ‚Äî Regista MediaService com foregroundServiceType + stopWithTask
- **`nowPlayingService.ts`** ‚Äî Comentario atualizado
- **`constants.ts`** ‚Äî Removida constante `RADIO_LOCK_SCREEN_DELAY` (nao utilizada)

### Ficheiros removidos

- **`patches/expo-audio+55.0.13.patch`** ‚Äî Eliminado. Nunca mais patchear expo-audio.

## Porque funciona (e a v1.14.0 falhou)

A v1.14.0 tentou criar um servico foreground proprio mas falhou porque expo-audio precisava de `setActiveForLockScreen(true)` para iniciar o seu `AudioControlsService`. Sem esse servico, o Android matava o audio apos ~3 minutos.

A v1.16.0 resolve isso porque o nosso `MediaService` e um foreground service independente. Qualquer foreground service no processo mantem TODOS os componentes vivos ‚Äî incluindo o ExoPlayer do expo-audio. O expo-audio nao precisa do seu proprio servico.

---

## O que e eliminado para sempre

- Race conditions entre expo-audio e o nosso modulo
- `recoverBuilder()` fragil para encontrar notificacao
- 6 retries (300ms ‚Üí 5s) para "apanhar" a notificacao
- Patch do expo-audio a cada atualizacao de SDK
- Polling para detetar play/pause do lock screen (agora sao eventos nativos)

---

## Fixes pos-auditoria (incluidos nesta release)

### Fix: Sincronizacao de notificacao
- `loadBitmap` agora suporta HTTP URLs (nao so file://). Quando o logo nao esta em cache, o service faz download com timeout 8s
- Fila de pending no modulo nativo: chamadas a `updateMetadata`/`updatePlaybackState` antes do service estar pronto sao enfileiradas e flushed via `onReadyCallback`
- Cache de dedup resetado apos `activate()` no radioService ‚Äî previne que updates legitimos sejam filtrados

### Fix: Auditoria critica
- **Threading**: `updateSessionMetadata()` e `postNotification()` sempre na main thread (antes corriam no artworkHandler thread ‚Äî violacao de thread safety)
- **Null intent**: `onStartCommand` chama `goForeground()` em intents null ‚Äî previne `ForegroundServiceDidNotStartInTimeException`
- **Bitmap lifecycle**: Intermediarios (scaled, pre-opaque) reciclados corretamente na cadeia `loadBitmap‚Üíscale‚ÜíensureOpaque` ‚Äî previne OOM apos muitas mudancas de musica
- **HTTP safety**: `downloadBitmap()` com `HttpURLConnection.disconnect()`, `inputStream.use{}`, rejeicao de ficheiros >5MB
- **onReadyCallback timing**: Deferred ate APOS artwork carregar (ou falhar)
- **MediaSession.isActive**: Deferred para `goForeground()` (apos transportCallback estar wired)
- **sendEvent try-catch**: Protege contra crash se module context destruido
- **Test mock**: Removido `overrideNotificationArtwork`, corrigidos nomes dos listeners
