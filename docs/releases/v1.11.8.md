# Release v1.11.8 (Build 56)

**Data**: 2026-04-19
**versionName**: 1.11.8
**versionCode**: 56 (Android)

---

## Resumo da Release

Correcao definitiva da atualizacao de artwork ao regressar do background. A v1.11.7 usou `recyclingKey` do expo-image que opera no nivel nativo (Glide) — insuficiente porque o Glide pausa requests em background e nao recarrega no `onResume`. Esta versao usa React `key` que forca unmount/remount do componente Image, contornando o lifecycle do Glide.

---

## Causa Raiz

O expo-image usa **Glide** como biblioteca de imagens no Android. Quando o app vai para background:

1. Glide **pausa** todos os requests de imagem (vinculado ao lifecycle da Activity)
2. No `onResume`, Glide retoma requests pendentes mas **NAO recarrega** imagens que ja foram loaded
3. O metodo `rerenderIfNeeded()` compara `sourceToLoad != loadedSource` — se a view nativa tem a imagem antiga carregada, o Glide ve `loadedSource` preenchido e nao recarrega mesmo com nova source

Referencia: [expo/expo #22100](https://github.com/expo/expo/issues/22100)

### Por que `recyclingKey` (v1.11.7) falhou

`recyclingKey` opera no nivel nativo do expo-image: seta `clearViewBeforeChangingSource = true` e chama `rerenderIfNeeded()`. Mas como o Glide esta em estado pausado/resumido apos background, a chamada pode ser ignorada. O `recyclingKey` foi projetado para FlashList recycling, NAO para transicoes de lifecycle.

### Por que `key` (React) funciona

O `key` opera no nivel do **React**, nao do modulo nativo:
1. React **destroi** o componente Image antigo (native view destruida pelo React reconciler)
2. React **cria** uma nova instancia do componente Image
3. Nova `ExpoImageViewWrapper` → `loadedSource = null` → Glide faz load **obrigatorio**

Validado pelo projeto web irmao (`olha-que-duas`), que usa `key={layerSrc}` no `<img>` para forcas remount no componente `ArtCrossfade`.

## Correcoes

### 1. NowPlaying.tsx — `key` em vez de `recyclingKey`

```tsx
// ANTES (v1.11.7):
<Image recyclingKey={musicRecyclingKey} source={{ uri: musicArtUri }} ... />

// DEPOIS (v1.11.8):
const songKey = `${nowPlaying.song.title}\0${nowPlaying.song.artist}`;
<Image key={songKey} source={{ uri: musicArtUri }} ... />
```

O `key` e baseado na **identidade da musica** (titulo + artista), nao na URI. Isto evita duplo remount quando `localArtUri` chega para a mesma musica (transicao remote → file://).

Aplicado nos tres modos com artwork: `music`, `podcast`, `announcement`.

### 2. useNowPlaying.ts — AppState listener (mantido da v1.11.7)

O listener de AppState que forca `setData({ ...current })` ao regressar do foreground continua a garantir que o React re-renderiza o NowPlaying com dados frescos e nova referencia de objeto.

## Comportamento por Cenario

| Cenario | key muda? | Resultado |
|---|---|---|
| Musica diferente em background | Sim | React unmount/remount → Glide load novo |
| 3+ musicas em background | Sim (ultimo songKey) | Mostra musica actual |
| Mesma musica, URI remote→file:// | Nao | expo-image atualiza source in-place |
| Mesma musica repetida | Nao | Sem flash desnecessario |
| localArtUri null (download falhou) | Sim (nova musica) | Usa remote URL (prefetched) |
