# v1.15.0 — Artwork de lock screen race-free

**Build 66** | 2026-04-25

## Problema

O artwork da notificacao/lock screen nao atualizava em background. A arquitetura anterior tinha uma race condition: o expo-audio recebia uma URL de artwork diferente a cada track change, iniciava um download async, e ao terminar re-postava a notificacao — sobrescrevendo o artwork que o modulo nativo tinha colocado. Em background, os 3 retries (300/800/1500ms) na main thread nao eram suficientes porque o Android desprioritiza a main thread.

## Solucao

Usar o bug do expo-audio (`URL.equals()`) como feature em vez de tentar contorna-lo.

### Nova arquitetura

1. **radioService.ts** — sempre passa o logo estatico da radio ao `setActiveForLockScreen`. Como a URL nunca muda, o expo-audio usa o bitmap em cache e nunca inicia download async. Metadata (title/artist) e artwork tem dedup independente.

2. **ExpoMediaSessionModule.kt** — gere o artwork da notificacao exclusivamente:
   - Attempt imediato + 6 retries (300ms-5s) numa HandlerThread dedicada (nao a main thread)
   - Sem `recycle()` de bitmaps (previne crash no Android 15)
   - Fundo opaco forcado em bitmaps com transparencia (fix overlap no Android 13)

3. **artworkCache.ts** — removido o file pool (`getLockScreenArtUri`, `lockScreenPool`, `lockscreen/` dir). Nao e mais necessario porque o expo-audio nao gere artwork.

### Porque e race-free

O expo-audio nunca inicia um download async de artwork (a URL do logo e sempre a mesma, cached via `URL.equals()`). O modulo nativo tem ownership exclusivo do artwork da notificacao.

## Limpeza

- Removido `MediaPlaybackService.kt` (386 linhas — foreground service legado nao utilizado)
- Removidas funcoes no-op do modulo (`updateMetadata`, `updatePlaybackState`, remote event listeners)
- Removida dependencia `androidx.media:media:1.7.0` (era usada pelo MediaPlaybackService)
- Removida entrada do `MediaPlaybackService` no AndroidManifest
- Removidos artefatos de build antigos do modulo nativo
