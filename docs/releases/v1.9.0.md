## Resumo da Release

Release com **dois eixos** focados em "agora a tocar":

1. **SSE em vez de polling** ‚Äî substituicao do polling de 3s pelo endpoint Server-Sent Events do AzuraCast/Centrifugo. O servidor empurra a troca de musica no instante exato em que acontece no stream. Polling de 3s mantido como fallback automatico
2. **Classificacao em 5 estados** ‚Äî o `nowPlayingService` deixou de classificar so "musica ou nada" e agora distingue **musica / programa ao vivo / podcast / anuncio-destaque / idle**, alinhando-se com a logica ja existente no site web (`olha-que-duas/src/hooks/useNowPlaying.ts`). Resolve o caso em que podcasts e eventos especiais nao mostravam capa nem nome no app

## Now Playing - SSE em vez de Polling

### O problema (residuo da v1.8.0)

Na v1.8.0 reduzimos o polling de 5s para 3s e removemos a janela de "transition" de 1.5s. Isso ja deu ~50% de melhoria, mas ainda restava uma janela de **0 a 3 segundos** entre o stream realmente trocar de musica e o app detectar (porque so descobriamos no proximo poll).

### A solucao

AzuraCast 6.x expoe um endpoint SSE via Centrifugo:

```
GET /api/live/nowplaying/sse?cf_connect={"subs":{"station:<shortcode>":{}}}
```

O servidor mantem a conexao aberta e empurra um frame `pub` cada vez que o `now_playing` muda no station. Latencia tipica do evento: **<200ms**.

### O que mudou no codigo

`src/services/nowPlayingService.ts` foi reescrito mantendo a mesma API publica (`start`, `stop`, `subscribe`, `cleanup`, `getCurrentData`). Internamente:

1. **Construtor**: extrai o `shortcode` dinamicamente da `streamUrl` (`/listen/<shortcode>/radio.mp3`) em vez de hardcoded. Constroi tanto a URL de polling quanto a URL de SSE a partir da mesma base
2. **`start()`**: dispara `fetchNowPlaying()` imediato + arranca polling fallback + tenta abrir SSE
3. **`connectSSE()`**: abre `EventSource` para `/api/live/nowplaying/sse?cf_connect=...`. No primeiro frame `connect` (handshake Centrifugo) ou `pub` valido, considera SSE conectado e **para o polling**
4. **Frames Centrifugo**:
   - `{"connect": {...}}` ‚Äî handshake, marca como conectado
   - `{"channel": "station:<shortcode>", "pub": {"data": {"np": {...}}}}` ‚Äî payload `np` tem a mesma forma do `/api/nowplaying/<shortcode>`, processado pelo mesmo metodo `processPayload`
   - Pings vazios e frames de outros canais sao ignorados
5. **Reconexao com backoff exponencial**: 1s -> 2s -> 4s -> ... -> max 30s. Reset do contador quando reconecta com sucesso
6. **Polling como fallback transparente**: se o SSE cair, o polling reinicia automaticamente para o utilizador nao perder updates enquanto a reconexao acontece em background
7. **AppState background**: fecha SSE para economizar bateria, polling vai para 15s (5x o `NOW_PLAYING_POLL_INTERVAL`). Foreground reabre SSE
8. **`cleanup()`**: cancela timers, fecha SSE, remove AppState listener

### Tipagem

Adicionadas interfaces minimas para os payloads externos (em vez de `any`):

- `AzuraNowPlayingPayload` ‚Äî `live`, `now_playing`, `song_history`
- `AzuraEntry` ‚Äî `played_at`, `duration`, `playlist`, `song`
- `CentrifugoFrame` ‚Äî `connect | channel | pub.data.np`

### Resultado

| Metrica | v1.7.0 | v1.8.0 | v1.9.0 |
|---|---|---|---|
| Deteccao de troca de musica | 0-5s (~2.5s medio) | 0-3s (~1.5s medio) | **<200ms** |
| Janela "limbo" mostrando radio | 1500ms fixo | 0ms | 0ms |
| Network 2a vez (cache imagem) | 200-800ms | 0ms | 0ms |
| Crossfade visual | hard cut | 350ms | 350ms |
| **Total perceptivel** | ~2-7s | ~1.5-3s | **~350ms** (apenas o crossfade) |

Em background, a estrategia anterior de polling lento (15s) e mantida ‚Äî SSE so corre em foreground.

### Resiliencia

- **Erro/timeout do SSE** -> polling fallback assume + tenta reconectar com backoff
- **Servidor fecha conexao** -> mesmo comportamento
- **Sem internet (offline)** -> SSE vai falhar; polling fallback vai falhar tambem; quando voltar online, ambos retomam (polling reentra a cada intervalo, SSE no proximo agendamento de reconnect)
- **`react-native-sse` 1.2.x** ‚Äî pure JS sobre XMLHttpRequest, sem modulo nativo, compatible com Expo SDK 54

## Now Playing - Classificacao em 5 Estados

### O problema

A versao anterior do `nowPlayingService` so reconhecia "musica valida" ou "nao musica". Quando a radio entrava em conteudo nao-musical legitimo (podcasts, eventos especiais, anuncios com arte propria) que vinha em playlists tipo `Podcasts`, `Especiais` ou `Eventos`, o filtro de jingle deitava tudo fora e o app caia para o logo da radio ‚Äî perdendo a capa e o nome do conteudo.

O site web (`olha-que-duas`) ja tinha uma classificacao mais rica em `src/hooks/useNowPlaying.ts`. Esta release porta essa logica para o mobile.

### Os 5 modos

O `NowPlayingData` ganhou um discriminador `mode: 'music' | 'liveShow' | 'podcast' | 'announcement' | 'idle'`. A deteccao acontece nesta ordem de prioridade dentro de `processPayload()`:

1. **`liveShow`** ‚Äî `data.live.is_live === true`. Streamer no microfone tem prioridade absoluta. Renderiza icone de microfone + `streamer_name`
2. **`music`** ‚Äî passa em `isValidSong()` (titulo/artista validos, duracao >= 60s, nao bate em `JINGLE_PATTERNS` nem em `JINGLE_PLAYLISTS`). Renderiza capa + titulo + artista
3. **`podcast`** ‚Äî playlist NAO bate em `MUSIC_PLAYLIST_PATTERNS` (e.g., nao tem "hits", "rotation", "manh√£", etc.) **E** duracao >= 60s **E** nao tem padrao de jingle no titulo. Renderiza capa do podcast + nome da playlist
4. **`announcement`** ‚Äî playlist bate em `ANNOUNCEMENT_PLAYLIST_PATTERNS` (`/an√∫ncio|especial|destaque|aviso|evento/i`) **E** nao tem padrao de jingle no titulo. **Sem restricao de duracao** ‚Äî spots de 15-30s com arte propria sao validos. Renderiza capa + nome
5. **`idle`** ‚Äî nada classificavel. Renderiza nome da radio + tagline (mesmo que antes)

### Buffer do ouvinte (`pickAudibleEntry`)

A API AzuraCast diz "o que o servidor esta a transmitir AGORA". Mas o ouvinte ouve com **~4s de atraso** (icecast burst de 2-3s @ 192kbps + decoder buffer de 3-5s). Antes de SSE isto nao era problema porque o polling ja chegava ~1.5s atrasado. Com SSE chegando em <200ms, a UI ficaria 4s adiantada do audio.

Solucao portada do web: `pickAudibleEntry()` percorre `[now_playing, ...song_history]` e devolve a entrada cuja janela `[played_at, played_at + duration)` contem `Date.now() - LISTENER_BUFFER_SECONDS`. Resultado: a UI mostra a musica que o ouvinte esta **realmente** a ouvir naquele momento.

### Hold-on-gap (8s)

O `song_history` do AzuraCast nao inclui jingles/vinhetas. Entre duas musicas ha tipicamente uma lacuna de 5-15s. Antes da release, a UI piscava para o logo nessa lacuna. Agora:

- Se a ultima entrada valida (musica/podcast/anuncio) terminou ha menos de `HOLD_PREVIOUS_ON_GAP_SECONDS` (8s), `processPayload()` ignora a lacuna e mantem o estado actual
- Se a lacuna excede 8s, cai para `idle`

Eliminacao do "piscar" entre faixas.

### Smart re-emit timer

Como o SSE so empurra quando a API muda, e a UI depende do `LISTENER_BUFFER`, em casos de borda (ex: track muito longa onde so chega uma SSE message) precisamos re-classificar localmente quando o ouvinte transitar para a proxima faixa. Solucao: `scheduleSmartReemit()` agenda um timer one-shot para `(played_at + duration - listenerWallClock + 1s) * 1000`. Quando dispara, re-roda `processPayload()` com `lastPayload` e `Date.now()` actualizado, recalculando o `audible`.

Em modo polling/SSE normal, este timer raramente dispara ‚Äî a proxima mensagem chega antes. Mas funciona como rede de seguranca.
