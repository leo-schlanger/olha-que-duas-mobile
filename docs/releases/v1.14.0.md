# Release v1.14.0 (Build 64)

**Data**: 2026-04-23
**versionName**: 1.14.0
**versionCode**: 64 (Android)

---

## Resumo da Release

Substituicao completa da notificacao/lock screen do expo-audio por um
**modulo nativo custom** (`expo-media-session`) que usa `MediaSessionCompat`
do Android ŌĆö o mesmo mecanismo que Spotify e YouTube Music usam.

Resolve definitivamente o problema de artwork que nao atualizava em
background quando a musica mudava. Elimina os 3 plugins custom
(withMetadataFix, withWifiLock, withStopAudioOnTaskRemoved) e a camada
de file copying do lock screen.

---

## O problema (residuo da v1.13.1-v1.13.4)

Desde a v1.13.1, o app tentou resolver o artwork congelado no lock screen
com varias abordagens:

| Versao | Tentativa | Resultado |
|--------|-----------|-----------|
| v1.13.1 | Fragment cache-busting (`#timestamp`) | Falhou ŌĆö `URL.equals()` ignora fragments |
| v1.13.2 | Patch nativo via `withMetadataFix.js` | Falhou ŌĆö AAR pre-compilado, patch nao compila |
| v1.13.3 | Refactor do `setActiveForLockScreen` | Falhou ŌĆö mesmo caminho nativo quebrado |
| v1.13.4 | File copying com path unico (`art-{ts}.jpg`) | Parcial ŌĆö funciona em foreground mas nao em background |

### Porque a v1.13.4 falhava em background

Dois problemas raiz identificados:

1. **Race condition no file copying**: `getLockScreenArtUri()` deletava o
   ficheiro anterior ANTES de copiar o novo. O lado nativo (thread de fundo
   com prioridade baixa em background) podia estar a ler o ficheiro antigo
   no momento em que era deletado ŌåÆ `BitmapFactory.decodeStream()` retornava
   null ŌåÆ notification sem artwork

2. **Chain de callbacks nativo fragil**: `setActiveForLockScreen()` ŌåÆ
   `setPlayerOptions()` ŌåÆ `loadArtworkFromUrl()` ŌåÆ callback ŌåÆ rebuild
   notification. Esta chain depende de `URL.openConnection()` + callback
   assincrono, e o rebuild da notificacao so acontece dentro desse callback.
   Em background, atrasos na thread nativa podem impedir o rebuild

Ambos os problemas sao **inacessiveis do lado JS** ŌĆö estao dentro do AAR
pre-compilado do expo-audio.

---

## A solucao: modulo nativo custom `expo-media-session`

Em vez de contornar o expo-audio, criamos o nosso proprio modulo que faz
a gestao da notificacao e MediaSession de forma independente:

```
expo-audio (playback only)          expo-media-session (notification)
  createAudioPlayer()                 activate() ŌåÆ foreground service
  player.play() / pause()            updateMetadata() ŌåÆ notification rebuild
  shouldPlayInBackground: true        updatePlaybackState() ŌåÆ play/pause icon
                                      onRemotePlay/Pause/Stop ŌåÆ JS events
```

### Porque funciona

| Aspecto | expo-audio (antes) | expo-media-session (agora) |
|---------|-------------------|---------------------------|
| Artwork | `URL.openConnection()` com `URL.equals()` bug | `BitmapFactory.decodeFile()` direto |
| Notification rebuild | Callback chain fragil | `NotificationManager.notify()` incondicional |
| File copying | Necessario para cache-bust URLs | Desnecessario ŌĆö native le direto do covers cache |
| MediaSession | Simplificada, parcial | `MediaSessionCompat` completa com `MediaSession.Callback` |
| WiFi lock | Plugin custom patchando Kotlin | `WifiManager.WIFI_MODE_FULL_HIGH_PERF` no service |
| Stop on swipe | Plugin custom no manifest | `onTaskRemoved()` no service |
| Lock screen controls | Polling de 5s para detectar | `MediaSession.Callback.onPlay/onPause` nativo |

## O que mudou no codigo

### Novo: `modules/expo-media-session/`

Modulo Expo local com dois ficheiros Kotlin:

**`MediaPlaybackService.kt`** ŌĆö Foreground service Android:
- `MediaSessionCompat` com `MediaSession.Callback` (play/pause/stop do lock screen e Bluetooth)
- `MediaStyle` notification com `NotificationCompat.Builder`
- Artwork carregado como `Bitmap` via `BitmapFactory.decodeFile()` ŌĆö sem network, sem URL parsing
- Bitmap escalado a max 512x512 para evitar `TransactionTooLargeException`
- `WifiManager.WifiLock` adquirido no `onCreate()`, libertado no `onDestroy()`
- `onTaskRemoved()` ŌĆö para tudo quando user faz swipe-to-kill
- Notification channel `olhaqueduas-radio` com `IMPORTANCE_LOW` (silencioso)
- `START_NOT_STICKY` ŌĆö nao reinicia se o sistema matar o service

**`ExpoMediaSessionModule.kt`** ŌĆö Ponte Expo Modules API:
- `activate(title, artist, artworkUri)` ŌĆö inicia o foreground service via `startForegroundService()`
- Bind ao service via `ServiceConnection` para chamadas diretas (sem Intent overhead)
- `updateMetadata(title, artist, artworkUri)` ŌĆö chamada direta via Binder
- `updatePlaybackState(isPlaying)` ŌĆö atualiza icone play/pause
- `deactivate()` ŌĆö unbind + `stopService()`
- Events `onRemotePlay`, `onRemotePause`, `onRemoteStop` ŌĆö wired via callbacks estaticos no service

**`src/index.ts`** ŌĆö API TypeScript:
- `activate(meta)`, `updateMetadata(meta)`, `updatePlaybackState(isPlaying)`, `deactivate()`
- `addOnRemotePlayListener()`, `addOnRemotePauseListener()`, `addOnRemoteStopListener()`
- Tipos `MediaMetadata` e `EventSubscription` exportados

### Modificado: `src/services/radioService.ts`

Reescrito para usar `ExpoMediaSession` em vez de `player.setActiveForLockScreen()`:

- **Removido**: `updateLockScreen()`, `resetLockScreenCache()`, `lastLockScreenMetaKey`, `lockScreenTimeout`, `clearLockScreenTimeout()`
- **Adicionado**: `activateMediaSession()`, `deactivateMediaSession()`, `updateMediaNotification()`, `lastMediaMeta` (dedup key), `mediaSessionActive` flag
- **Adicionado**: `setupRemoteCommandListeners()` ŌĆö subscreve `onRemotePlay/Pause/Stop` do modulo nativo. Play/pause do lock screen agora sao tratados via callback nativo em vez de polling de 5s
- **`play()`**: chama `ExpoMediaSession.activate()` em vez de `setActiveForLockScreen(true)` com timeout de 200ms
- **`pause()`**: chama `ExpoMediaSession.updatePlaybackState(false)` em vez de nada (antes o polling detectava)
- **`stop()`**: chama `ExpoMediaSession.deactivate()` em vez de `setActiveForLockScreen(false)`
- **`subscribeToNowPlaying()`**: usa `data.localArtUri` (do covers cache) diretamente ŌĆö sem `getLockScreenArtUri()` file copying
- Interface publica **100% identica** ŌĆö `useRadio`, `useNowPlaying`, `RadioPlayer`, `NowPlaying` nao precisaram de alteracao

### Simplificado: `src/utils/artworkCache.ts`

- **Removido**: `LOCKSCREEN_DIR_NAME`, `lockScreenDir`, `lastLockScreenFile`, `ensureLockScreenDir()`, `getLockScreenArtUri()`
- **Mantido**: `getLocalArtwork()`, `getCachedArtwork()`, `prefetchLogo()`, `getLogoUri()`, `pruneCache()`
- O covers cache (`olhaqueduas-covers/`) continua intacto ŌĆö o modulo nativo le diretamente dele

### Modificado: `App.tsx`

- **Removido**: `import { preload as preloadAudio } from "expo-audio"` e chamada `preloadAudio()`
- **Mantido**: `prefetchLogo()` (necessario para o artworkUri do modulo nativo)

### Removido: 3 plugins custom

- `plugins/withMetadataFix.js` ŌĆö desnecessario (bitmap direto, zero `URL.equals()`)
- `plugins/withWifiLock.js` ŌĆö WiFi lock agora no `MediaPlaybackService`
- `plugins/withStopAudioOnTaskRemoved.js` ŌĆö `onTaskRemoved()` agora no `MediaPlaybackService`
- `app.json` ŌĆö removidas as 3 entradas de plugins

### Mantido sem alteracao

- `src/services/nowPlayingService.ts` ŌĆö SSE + polling + classificacao 5 estados (inalterado)
- `src/hooks/useRadio.ts` ŌĆö interface identica
- `src/hooks/useNowPlaying.ts` ŌĆö interface identica
- `src/components/RadioPlayer.tsx` ŌĆö sem alteracoes
- `src/components/radio/NowPlaying.tsx` ŌĆö sem alteracoes
- `expo-audio` ŌĆö mantido no `package.json` para playback de audio (sem lock screen)
